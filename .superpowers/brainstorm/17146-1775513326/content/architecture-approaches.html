Timer Architecture: Zustand Store Design

How the global timer store coordinates with the server

A

Zustand as Cache Layer (Recommended)

Zustand owns runtime timer state. React Query still fetches habits, but timer state lives in zustand — not derived from the habits query.

Pros

  • Instant UI updates — no waiting for query invalidation
  • Timer state decoupled from habit fetching
  • Easy to read from any component (mini-bar, habits view, etc.)
  • Hydrate once on app load from habits query, then zustand is source of truth

Cons

  • Two sources of truth — must keep zustand + DB in sync
  • Need hydration logic on page load

Flow: Click Start → zustand updates immediately (optimistic) → API call fires → on success, store confirmed. On failure, rollback store.

B

React Query Primary, Zustand as Derived Mirror

React Query remains the source of truth. Zustand subscribes to the query cache and mirrors the active timer state for convenient access.

Pros

  • Single source of truth (React Query cache)
  • No sync issues — zustand always reflects query state
  • Existing invalidation patterns keep working

Cons

  • Timer updates still depend on query invalidation latency
  • Doesn't fully solve the original desync problem
  • Zustand adds complexity without much benefit over just reading the cache
C

Zustand Only (Drop React Query for Timer)

Timer operations go through zustand actions that call the API directly. React Query handles habits/sessions/rankings but not the active timer.

Pros

  • Cleanest separation — timer is fully independent
  • No dual-cache confusion
  • Zustand actions = simple async functions

Cons

  • Must manually trigger React Query invalidation for habits after stop
  • Active timer data currently embedded in habits response — need to restructure
